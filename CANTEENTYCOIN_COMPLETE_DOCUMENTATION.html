1. System Overview

CanteenTycoon AI is a browser-based desktop operating system simulation for school cafeteria food waste management. It combines a LightGBM machine learning model (trained on 2,600+ real university canteen records) with Claude 3.5 Sonnet as an intelligent orchestrator to predict, analyze, and reduce food waste.

Core Principles:

  • Zero hardcoded data — Every prediction is driven by the LightGBM model + live weather + calendar
  • Claude as orchestrator — Claude routes intents, calls tools, generates predictions, and creates PDFs of anything
  • Visual simulation is decorative — NPCs walking around and simulated waste events are visual-only; Claude uses only real data (ML model, weather, calendar)
  • Proxy architecture — All external APIs route through a local Python proxy server to avoid CORS issues

Tech Stack: HTML5 Tailwind CSS Vanilla JS (3,166 lines) Python (328 lines) Flask LightGBM GSAP (animations) · Lucide (icons) · Claude 3.5 Sonnet · Gemini 2.5

3.4 js/npc.js — NPC Simulation (191 lines)

Classes:

  • StudentNPC — 5 types (Regular, Vegetarian, Eco, Athlete, Picky) with 8-state lifecycle
  • ManagerNPC — 3 modes (patrol, phone, idle)

NPC State Machine:

SPAWNING → ENTERING → AT_COUNTER → FINDING_SEAT → EATING → ABANDONING → LEAVING → (repeat)
                              (queue 5-20 ticks)          (waste prob)    (to bin)

Important: NPC waste events are visual-only. They increment wasteToday but Claude is explicitly instructed to ignore simulated data. All AI predictions use only the LightGBM model + real weather + calendar.

3.8 app.py — ML Server (52 lines)

POST /predict { Date, Meal, Canteen_Section, Food_Category }
    → Parse JSON → Extract date features (Year, Month, Day, DayOfWeek, WeekOfYear)
    → One-hot encode categoricals → Reorder columns → LightGBM regressor
    → Return { predicted_waste_kg }

Model: LightGBM trained on 2,600 university canteen records. Features: 6 numerical (Cost, Year, Month, Day, DayOfWeek, WeekOfYear) + one-hot encoded (Meal, Section, Category). Files: lgbm_food_waste_predictor.joblib (280KB), one_hot_encoder.joblib (1.5KB).

6. Key Design Decisions

DecisionRationale
Proxy architecture for all APIsEliminates CORS issues entirely. Browser talks to same origin (localhost:8080), server forwards to external APIs. Also enables timeout control (180s for NIM, 15s for ML, 10s for DuckDuckGo).
ML model feeds Claude, doesn't replace itLightGBM provides a data-driven baseline (2.07 kg). Claude adjusts for weather (rain → +waste), calendar (exam week → −attendance), menu type, and special events. Final decision is Claude's.
Simulated data purged from AI contextNPC waste events are visual-only. Including them in prompts caused Claude to report "126.90kg waste" from simulated NPCs. Claude now receives only: ML baseline + weather + calendar + menu + hours.
Claude as intent routerInstead of regex keyword matching, Claude classifies each message into FORECAST/CHAT/SIMULATION_CONTROL/DATA_QUERY with structured JSON output. This makes intent detection context-aware and extensible.
GSAP for all animationsProvides consistent, performant animations with spring physics (back.out, elastic.out) and timeline sequencing. Used for welcome gate, window transitions, prediction paper bars, widget counters, and card stack.
Zero hardcoded dataHISTORY starts empty. NIM builds everything from live observation + reference dataset. The real dataset (food_waste.json) is reference-only — never merged into session HISTORY.
Google OAuth graceful degradationWhen no client ID configured, the app shows a link to settings. App is unverified, so only test users work. Calendar sync shows clear error messages (403 → "enable Calendar API").
Icons have zero-CDN dependencyIf Lucide CDN fails, a fallback script replaces all [data-lucide] elements with emoji from emojiMap. All dynamically created windows use raw emoji instead of Lucide.

7. How to Run & Deploy

Local Development

# Terminal 1: ML API
python app.py                           # → localhost:5000

# Terminal 2: Web + Proxy Server
python server.py                        # → localhost:8080

# OR use the convenience script:
./start.ps1                             # Launches both in hidden windows

Browser

Open http://localhost:8080
1. Configure API keys in Settings (or edit api.env)
2. Fill wizard (4 cards) → upload photo (optional)
3. Click Continue → desktop boots
4. Chat with Claude: "predict tomorrow", "create a pdf menu", etc.

API Keys Required

KeyWherePurpose
Claude API Key (Anthropic)Settings → ClaudePrimary AI: intent routing, predictions, chat, PDF content
Gemini API Key (Google)Settings → GeminiMap generation (image), vision analysis, fallback for chat
Google Client IDSettings → GoogleOAuth sign-in + Calendar sync (optional)

Deployment (Render.com)

1. Push to private GitHub repo
2. Create Web Service: `python server.py`
3. Create separate service: `python app.py` (ML API)
4. Set environment variables in Render dashboard
5. Update proxy URLs in config.js for production
